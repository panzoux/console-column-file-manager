Preview pane — edge cases & adaptive layout

How the preview behaves at different terminal widths and minimum sizes

a) File at the rightmost column — preview replaces the "would-be next" slot

When a file is selected, columns are pruned to the right. The preview takes slot N+1 (32 chars) + all remaining. This means when preview is active, we show one fewer dir column to make that slot available.

visibleDirCols = floor(termWidth / colWidth) − 1
previewWidth = termWidth − (visibleDirCols × colWidth)

b) Width breakpoints

≥ 96 wide — comfortable (e.g. 120: 2 dir cols + 56-char preview)
│ C:\ │ Users │ clip.mp4 MP4 Video • 87 MB │ │ Desktop/ │ > clip.mp4 │ Modified: 2026-02-10 18:55 │ │ readme.txt │ photo.jpg │ Dimensions: 1920 × 1080 px │ │ photos/ │ music/ │ Duration: 00:03:42 │
64–95 wide — narrow (e.g. 64: 1 dir col + 32-char preview)
│ Users │ clip.mp4 MP4 • 87 MB │ │ > clip.mp4 │ 1920 × 1080 px │ │ photo.jpg │ Duration: 00:03:42 │ │ music/ │ │
< 64 wide — preview hidden automatically (not enough room for 1 col + useful preview)
│ Users │ Desktop/ │ │ Desktop/ │ readme.txt │ │ > readme.txt │ │ │ photos/ │ photos/ │

Preview hidden silently — Shift+V state is preserved, it re-appears when terminal widens. Status bar shows a hint: [preview hidden: narrow terminal]

c) Adaptive hex dump — bytes-per-row scales with available width

Formula: bytesPerRow = max(4, (previewWidth − 14) / 4) — address (8) + spacing (2) + per-byte (3) + ASCII sep (2) + per-byte ASCII (1) = 14 + 4N

32 wide → 4 bytes/row

56 wide → 10 bytes/row

d) Column width is an input, not a constant

Preview width calculation never hardcodes 32. It sums the actual display widths of drawn columns:

previewWidth = terminalWidth − drawnColumnsDisplayWidth

Where drawnColumnsDisplayWidth is computed from whatever column widths are in use — works correctly when columns become resizable. All content-width calculations inside the preview renderer also take previewWidth as a parameter, never a constant.
